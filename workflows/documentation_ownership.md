# Documentation Ownership

## Purpose

Ensure documentation remains organized, current, and owned by the appropriate role.

Clear ownership prevents documentation decay and ensures that agents always operate with reliable information.

---

## Ownership Structure

Product Manager

Responsible for:

- product vision documents
- product strategy
- product requirement documents (PRDs)
- roadmap
- milestone definitions
- market analysis

Location:

/company  
/product

---

Technical Architect

Responsible for:

- system architecture documentation
- architecture decision records
- coding standards
- repository structure standards
- security principles
- infrastructure design documentation

Location:

/architecture

---

Project Manager

Responsible for:

- workflow documentation
- sprint process documentation
- release processes
- task coordination documentation
- operational execution procedures

Location:

/workflows

---

Quality Control Analyst

Responsible for:

- testing standards
- code review standards
- QA procedures
- quality enforcement guidelines

Location:

/workflows

---

Software Engineers

Responsible for:

- updating documentation when implementation changes behavior
- keeping inline documentation accurate
- ensuring example usage remains correct

Engineers should not define architecture standards but must follow them.

---

## Documentation Update Rules

Documentation must be updated when:

- architecture changes
- system behavior changes
- APIs change
- workflows change
- testing standards change

Failure to update documentation creates long term system risk.

---

## Documentation Quality Standards

Documentation should be:

Clear  
Concise  
Accurate  
Easy to navigate  

Avoid:

- unnecessary verbosity
- outdated information
- duplicated documentation across multiple files
- keep it stupid simple so a 5th grader would understand

Prefer one authoritative source per topic.
